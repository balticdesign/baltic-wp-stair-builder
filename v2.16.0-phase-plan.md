## Locked decisions

1. **Two availability mechanisms** (brief §3.4). `available_for` multi-select tags on
   material/profile rows (values = construction-type codes) **plus** `strict_for` multi-select on
   the construction type naming which repeaters it gates strictly. Permissive by default: untagged
   rows stay available unless the repeater is in `strict_for`. Empty `available_for` = "available to
   all" handles only *subtractive* rules — `strict_for` is required for the *restrictive* ones.
2. **Regime overrides global** Construction Settings where it supplies a value, **including hard
   maximums** — a regime minimum must be able to raise a global hard-max out of the way (brief §2,
   verify item 6). Global remains the fallback.
3. **`display_total_run` is separate from `total_run`.** `total_run` stays byte-identical (feeds
   pricing). The 43mm top lip only ever touches `display_total_run` + the canvas (brief §5.2).
4. **Commercial splits into two ADK classifications** — General Access and Utility — **both
   customer-selectable** (brief §3.2). Labels plain, ADK terms in the description.
5. **Seed-data test:** *"Would this value be wrong for the next licensee?"* (brief §11). ADK limits →
   no (seed, editable). Prices, names, supplier specs → yes (empty/neutral default).
6. **Lock `construction_type.code`** after creation (read-only on persisted rows; sanitiser
   preserves the stored value). Rationale: repeater rows have no stable row id, so a save cannot
   distinguish a *rename* from a *delete-then-recreate-at-index* — a cascade can't get a trustworthy
   old→new map. Locking is the only robust option and matches the v2.0.0 contract (code = stable
   machine key, name = editable label).
7. **Slugify NEW construction codes on save; NEVER retro-slug existing ones.** `construction_code`
   is currently unslugified free text, so new codes must be slugified (else `open_riser` tags won't
   match a hand-typed `Open Riser`). Retro-slugging an existing code would orphan historical leads
   via `bd_code_label` — do not.
8. **Orphan-drop on sanitise, made visible.** Deleting a construction type drops tags referencing
   its code; if that empties a strict repeater, surface the §6.1 admin warning
   (*"Open Riser has no valid riser types"*) — never let a construction type silently vanish.
9. **`tread_profile_code` is NOT locked in Phase 0** — unaudited, and a profile's own code is not a
   tag target (tags store construction codes). It joins the hardening release, audited as a set.
10. **Extending the lock to every repeater code is a separate hardening release**, not this epic
    (see *Follow-ups*). `stringer/tread/riser` codes are constrained material dropdowns — the code is
    a meaningful **editable attribute** (the material), not an identifier; locking it is the wrong
    tool.

**Versioning:** each phase is an independently-shippable **minor** bump, set **at commit time**
(one bump per commit, no point-bumps): Phase 0 → 2.16.0 … Phase 5 → 2.21.0.
**Sequencing:** phase-by-phase, commit + review after each. **Hardest pause after Phase 1 and Phase 4.**

| category | ships as | rows |
|---|---|---|
| **Facts about the world** — jurisdictional limits, identical for every UK client | Seeded, editable | The four `building_regs` rows (§3.2) + all numeric columns; 100mm sphere; 2R+G 550–700 |

not offered until the client configures their own boards (zero rows → strict filter empty → §6.1).

## Phases

### Phase 0 — Availability data model + code lock (2.16.0) ✅ BUILT
**Scope adjusted during build (flagged):** the `available_for`/`strict_for` FIELDS moved here from
Phase 2, and LOCALISATION moved to Phase 2. Rationale: a field type with no consumer isn't
verifiable, and localisation belongs with its consumer (the JS enforcement). Net split — Phase 0 =
availability **data model** (persists + validates in admin); Phase 2 = **enforcement** (front end).
- **Multi-select field type** (checkbox group), reusable — `render_multiselect` + shared
  `render_multiselect_group` + `resolve_field_choices` (dynamic `choices_source: construction_codes`
  or static `choices`) + `sanitize_multiselect`. Top-level dispatch + generic-card-row branch.
- **Fields:** `strict_for` (static choices: the 4 repeater ids) on `construction_types`;
  `available_for` (dynamic construction codes) on `stringer/tread/riser_types` + `tread_profiles`.
- **`SCHEMA_VERSION` 3 → 4** — additive (new sub-fields); **no migration** needed, missing keys
  default to empty. Bump justified by the shape change.
- **`construction_code`: slugify-new + lock** — `lock_code` flag on the repeater; persisted rows'
  code renders read-only; `slugify_new_codes()` slugifies NEW codes only and **preserves existing
  ones byte-for-byte** (never retro-slug → protects `bd_code_label` historical leads). The
  "already in the stored set?" test is the only safe rename/create signal without stable row ids.
- **Orphan-drop** — order-independent post-pass in `sanitize()`: drops `available_for` tags
  referencing a construction code absent from the final save. `strict_for` (static) validated in
  `sanitize_multiselect`. The paired §6.1 empty-strict-repeater admin warning is Phase 2 (needs the
  enforcement layer).
- **Verified** (dev, via `wp eval`): code preserved on rename; typed `open_riser` kept; blank code →
  `cut-string`; `bogus_repeater` dropped from strict_for; `ghost_code` dropped from available_for
  while a same-save `open_riser` survived; schema stamped 4; render smoke-test (readonly code +
  checkbox groups) clean.
- **Not here (deferred):** localisation, JS filtering, server revalidation → Phase 2.

### Phase 2 — Construction availability ENFORCEMENT (2.18.0) ✅ BUILT
Data model (tags + code lock) shipped in Phase 0; this phase enforces it. **Design signed off with
amendments; both directions + code+price row-matching + admin warning verified via `wp eval`. JS
filter DOM behaviour is DESIGN-verified only — browser not exercised (extension blocked).**

- **Row addressing without row identity:** each material/profile `<option>` carries
  `data-available-for` (from its row), emitted via `bd_stairbuilder_normalise_repeater` +
  `form-template.php`. The filter reads tags off the option, so the two non-unique `pine` rows are
  distinct at the DOM level. No payload/row-id change.
- **Localisation** `stairBuilderVars.availability` = `{ strict_for: {code:[repeaterIds]},
  repeater_select: {repeaterId:'#selectId'} }`. `available_for` rides on options.
- **Filter** (`formLogic.js`, new module): snapshot-at-init (confirmed the filter is the only select
  rebuilder — `setMaterial` only sets `selected`), rebuild per §3.4 (permissive: untagged OR includes;
  strict: must include), individual options removed (not disabled/CSS-hidden). Selection survives or
  falls back to first valid + `.sb-limit-msg` + fires `change`.
- **Single valid option → stated line, not a one-option select** (`.bd-avail-collapsed` hides the
  select — kept in DOM so it POSTs + priceCalc reads it unchanged; `.bd-avail-stated` shows the line).
  `data-bd-reason` distinguishes `filtered` (strictness removed the rest → "· required for X") from
  `catalogue` (client configured one row → no reason). Summary already reads the select value.
  Markup is minimal/styleable — **visual treatment goes through Dan's design pass.**
- **Construction type not offered when a strict repeater has zero tagged rows** (pruned from
  `#construction_type` at init) + **admin-facing warning** in settings (`empty_strict_warnings()`):
  *"Open Riser has no valid riser types…"* — never silent (§3c).
- **Server revalidation** (`bd_stairbuilder_revalidate_availability`, called after `parse_str`,
  before `create`, 422 on violation). BOTH directions (strict membership + permissive non-empty must
  include). Row addressing via an un-stripped `revalidate_meta` (code:price) — storage/PDF stay on
  stripped codes. **Coverage:** row-granular for unique codes AND wherever code+price disambiguates
  (proven: `pine:25` rejected, `pine:45` allowed under Open Riser). **GAP (documented at the code
  site + Baltic Mind):** two rows sharing BOTH code and price fall back to code-granular — deferred
  to the stable-row-identity release.
- **FULLY GENERIC — confirmed:** no construction code is special-cased in JS, PHP, or settings.
  `cut_string`/`open_riser` are the *codes SPD will use*, but nothing keys on them; a licensee with
  zero tags has empty `strict_for` + untagged rows, so every option shows and every combination
  passes — pre-v2.16 behaviour exactly.

## Corrections to the brief

- **Metal spindle count does NOT read `total_run`.** The brief (§5.2 item 1) says it does; it does
  not. `priceCalc.js:11-26` recomputes its own run from a fixed **240mm-going / 42°-rake** assumption
  (`rakePerStep = 240 / cos(42°)`, lines 187/220). **`calculateRake()` is the sole real consumer of
  `total_run`.** The §5.2 boundary holds either way — leave `total_run` byte-identical — but the
  inline comment must state what's *true*, not repeat the brief's claim.

- **Supersede "empty `available_for` = available to all" as sufficient on its own.** It expresses
  only *subtractive* rules. The client's rules are *restrictive*, so `strict_for` is required (brief
  §3.4). Recorded so we don't regress to a tag-only scheme.

## Findings

- **`tread_types` / `riser_types` codes are non-unique** — `pine` and `oak` each appear on multiple
  rows (different thicknesses) because the code sub-field is the v2.8.0 constrained material
  dropdown. **Consequences:**
  1. *Filtering:* `available_for` cannot key an option on its code (Phase 2 must use a row id).
  2. *Live correctness bug:* `bd_code_label()` (`stairbuilder_pdf.php:25`) resolves a stored code to
     the **first** matching row's name. A lead storing `pine` for a 30mm-pine tread renders as
     `Solid Pine (25mm)` on the quote PDF — **price correct, label wrong**. This exists today for
     every historical lead across these repeaters. → the hardening release is a **correctness fix**,
     not a nicety.

- **240/42 metal assumption is deliberate.** When going ≠ 240 the metal spindle count is computed
  for a staircase the customer isn't buying. This is intentional consistency with the live
  `baltic-wc-spindle-calculator` product page (customers compare the two). **Documented, not fixed.**
