# Stairbuilder 1.7.0 — Pan/Zoom Viewport

**Plugin version inspected:** `1.6.4`
**Target version:** `1.7.0`
**Date:** 2026-05-12
**Builds on:** 1.6.4 (responsive sizing profiles, straight-flight init race fix)

---

## TL;DR

Replace the current aspect-ratio-driven canvas (which grows tall enough to fit the worst-case staircase and leaves dead space below the diagram on most configurations) with a **fixed-height viewport** that the user can pan and zoom within. Default zoom fits the staircase to ~80% of viewport height. Wheel-zoom, pointer-drag-pan, pinch-zoom on touch. Reset-to-fit via a corner button or double-tap.

This makes the canvas behave the way TKstairs' canvas behaves on mobile and desktop — a sensible-sized window onto a navigable diagram — without rebuilding on Babylon or any other 3D engine (we stay in plain canvas 2D).

---

## 1. What's driving this

Two things you'll have noticed after 1.6.4 landed:

1. **Dead space below the diagram on desktop.** The canvas is `aspect-ratio: 568/506`, so at 1200px wide it reserves ~1069px of height. With the wide sizing profile shrinking the staircase to ~50% of canvas height, the bottom half of the canvas is empty. That dead space is now baked into page layout — users scroll past blank cream-coloured area to reach the quote panel underneath.

2. **The mobile case wants the opposite shape.** On mobile, viewport is narrow and tall. The current aspect-ratio gives us a tall narrow canvas where the staircase sits centred with empty space top and bottom. TKstairs' mobile UX fits the staircase to the full available viewport with pinch-zoom for detail — much more usable.

Both problems share a root cause: **the canvas tries to be a container that's always big enough for the staircase, when it should be a viewport onto a navigable scene.**

---

## 2. Architecture overview

### Coordinate spaces

After this change, the plugin has three coordinate spaces (currently has two):

| Space | Units | Used for |
|---|---|---|
| **Real-world** | mm | Form input values (`floor-height`, `going`, `stair-width`) |
| **Staircase-local** | px | What `drawTreads()`, `drawPosts()` etc draw. Positions like `treadHeight`, `Stairs.centerX`, `Stairs.maxHeight`. **Unchanged from current behaviour.** |
| **Canvas/screen** | px | Final pixels rendered. = staircase-local space × `viewport.zoom` + `viewport.pan`. **New.** |

The staircase-local space stays exactly as it is. We don't touch `drawTreads`, `drawPosts`, `drawBallustrades`, `drawMeasures` or any of the other 30+ draw helpers. The transform from staircase-local to canvas/screen is applied via `context.scale()` + `context.translate()` once, in `Stairs.draw()`.

### State

New `Stairs.viewport` object, defined once at module load alongside `Stairs.canvas`:

```js
Stairs.viewport = {
    zoom: 1,          // current scale factor — 1.0 = fit-to-viewport default
    panX: 0,          // pan offset in canvas-space px
    panY: 0,          // pan offset in canvas-space px
    fitZoom: 1,       // computed fit-to-viewport zoom (recalculated per draw)
    isDragging: false // pointer-drag state for input handler
};
```

**Crucial: `Stairs.viewport` is NOT a property of `Stairs.options`.** `Stairs.options` is rebuilt from form values every time `onLoad()` fires (e.g. user changes floor height). Putting viewport state inside `options` would reset the user's zoom/pan every keystroke. `Stairs.viewport` is module-scoped — it persists across form-driven redraws.

The only time `viewport` resets to defaults (`zoom: fitZoom, panX: 0, panY: 0`) is **when the staircase type changes** (user toggles straight/quarter/half flight). Detected by comparing `Stairs.options.type` against a remembered `Stairs._lastType`.

## 3. The fix — in order

### 3a. Canvas-container height

In `assets/css/layout.css`, replace the aspect-ratio rule with a fixed-height viewport:

```css
/* Before */
#canvas-container {
  /* ... */
  max-width: 1200px;
  aspect-ratio: 568 / 506;
}

/* After */
#canvas-container {
  /* ... */
  max-width: 1200px;
  height: min(70vh, 700px);   /* fixed viewport, never taller than 70% of screen */
  min-height: 360px;          /* mobile-portrait floor */
}
```

The `min()` function ensures: on desktop, capped at 700px regardless of viewport; on mobile portrait, 70% of viewport height (sensible touch target without dominating); on very small screens (e.g. landscape mobile), the 360px floor keeps the diagram usable.

**Delete the `aspect-ratio` declaration** — we're not constraining by ratio anymore.

### 3b. Canvas pixel sizing in layout.js

In `assets/js/layout.js`, `sizeCanvasToContainer()` currently derives height from width via `CANVAS_RATIO_H / CANVAS_RATIO_W`. After this change, height comes from the container's `clientHeight` instead. Replace lines 160–167 of `sizeCanvasToContainer()`:

```js
// Before
const w = Math.min(container.clientWidth || CANVAS_MAX_WIDTH, CANVAS_MAX_WIDTH);
if (!w) return;
const h = Math.round(w * (CANVAS_RATIO_H / CANVAS_RATIO_W));

container.style.height = h + 'px';
if (c.width === w && c.height === h) return;
c.width = w;
c.height = h;

// After
const w = Math.min(container.clientWidth || CANVAS_MAX_WIDTH, CANVAS_MAX_WIDTH);
const h = container.clientHeight;
if (!w || !h) return;

if (c.width === w && c.height === h) return;
c.width = w;
c.height = h;
// container.style.height is now CSS-controlled — don't set it from JS
```

The `CANVAS_RATIO_W` and `CANVAS_RATIO_H` constants at the top of `layout.js` become unused — leave them as commented-out historical reference rather than deleting outright, in case we ever need to revert.

### 3c. Viewport state on Stairs object

In `assets/js/Stairs.js`, after the existing `Stairs.canvas = null;` line (~line 47), add:

```js
// Pan/zoom viewport state. Lives outside Stairs.options because options is
// rebuilt from form values on every onLoad(); viewport must persist across
// form-driven redraws so the user's zoom level survives input changes.
// Reset (to fitZoom + zero pan) only when staircase type changes.
Stairs.viewport = {
    zoom:       1,
    panX:       0,
    panY:       0,
    fitZoom:    1,
    isDragging: false,
    _lastType:  null  // set by Stairs.init; triggers reset when type changes
};

// Zoom clamps. Below 0.3 the staircase is unreadable; above 5 individual
// tread lines disappear into pixel rounding. Cursor-anchored zoom is
// clamped against these every wheel/pinch event.
Stairs.ZOOM_MIN = 0.3;
Stairs.ZOOM_MAX = 5;
```

### 3d. Fit-to-viewport calculation

Add a new method that computes the zoom level needed to fit the staircase to 80% of viewport height. Place it after `setRegularStairsOptions()` / before `setQuarterturnStairsOptions()` so it sits with the other geometry helpers.

```js
// Computes the zoom factor that scales the staircase's bounding box to
// occupy ~80% of viewport height. Called once per draw, after the
// staircase-local geometry is computed but before context transforms apply.
//
// Uses height as the limiting dimension (per Dan's spec: viewport is
// always wider or square, never taller than wide for landscape desktop;
// mobile portrait is the only narrow-viewport case and the staircase
// width is typically the limiting factor there anyway — but we check both).
Stairs.computeFitZoom = function () {
    var stairHeight, stairWidth;

    switch (Stairs.options.type) {
        case Stairs.StairTypeEnum.REGULAR:
            stairHeight = Stairs.maxHeight;
            stairWidth  = Stairs.options.treads.width;
            break;
        case Stairs.StairTypeEnum.QUARTERTURN:
        case Stairs.StairTypeEnum.DOUBLETURN:
            // Stairs.widthPx and Stairs.maxHeight are the exact local-space
            // bounding-box dimensions for these turn types — already computed
            // in setQuarterturn/setDoubleturnStairsOptions. maxHeight includes
            // the turn flight's vertical contribution.
            stairHeight = Stairs.maxHeight;
            stairWidth  = Stairs.widthPx;
            break;
        case Stairs.StairTypeEnum.HALFTURN:
            // Half-turn uses maxHeight1 (per the existing HALFTURN draw case).
            stairHeight = Stairs.maxHeight1;
            stairWidth  = Stairs.widthPx;
            break;
        default:
            return 1;
    }

    if (!stairHeight || !stairWidth) return 1;

    var TARGET_FILL = 0.8;  // staircase fills 80% of the limiting viewport dimension
    var fitByHeight = (Stairs.canvas.height * TARGET_FILL) / stairHeight;
    var fitByWidth  = (Stairs.canvas.width  * TARGET_FILL) / stairWidth;
    return Math.min(fitByHeight, fitByWidth);
};
```

### 3e. Type-change detection and viewport reset

In `Stairs.init()` (~line 74), after `Stairs.canvas = canvas;` and before the redraw, add the type-change check. Use an explicit boolean flag, not a sentinel value on `zoom` — see note below.

```js
Stairs.init = function (canvas, config) {
    Stairs.canvas = canvas;
    Stairs.options = Stairs.applyOptions(config);

    // Detect staircase-type change and flag for viewport reset.
    var needsFitZoom = false;
    if (Stairs.viewport._lastType !== Stairs.options.type) {
        Stairs.viewport._lastType = Stairs.options.type;
        Stairs.viewport.panX = 0;
        Stairs.viewport.panY = 0;
        needsFitZoom = true;
    }

    // ... existing setRegular/Quarter/Half/DoubleturnStairsOptions call ...

    // After geometry is set, recompute fit zoom. Replace user zoom with
    // fit zoom only if a type change flagged a reset; otherwise preserve
    // whatever the user has pan/zoomed to.
    Stairs.viewport.fitZoom = Stairs.computeFitZoom();
    if (needsFitZoom) {
        Stairs.viewport.zoom = Stairs.viewport.fitZoom;
    }

    Stairs.draw(canvas);
};
```

**Why a flag, not a sentinel on `zoom`.** An earlier draft used `Stairs.viewport.zoom = 0` as a "use fit zoom" sentinel, replaced with the real fit zoom after geometry. That's racy: viewport.js's input handlers can fire between the sentinel assignment and the replacement (e.g. a pointermove queued during a slow init on mobile, or a wheel event from a fast scroller). `clampZoom(0 × factor) = ZOOM_MIN = 0.3`, the sentinel is lost, and the user sees zoom=0.3 instead of fit-zoom forever. A local boolean `needsFitZoom` removes the race window entirely — no shared state, no observable intermediate value.

### 3f. Apply the transform in Stairs.draw

The current `Stairs.draw()` has four `switch` cases (REGULAR, QUARTERTURN, HALFTURN, DOUBLETURN) that each follow the same shape: `clearRect → save → fillRect → restore → [save → translate → draw stuff → restore]`. We wrap the per-case draw block with our viewport transform.

For the REGULAR case specifically (which currently doesn't have a `save/translate/restore` wrapper — it draws directly), we add one. Pattern to apply in all four cases:

```js
case Stairs.StairTypeEnum.REGULAR: {
    var treads = Stairs.options.treads;
    var posts = Stairs.options.posts;
    var ballustrades = Stairs.options.ballustrades;

    // Clear and fill background — these MUST be outside the viewport
    // transform (we always want the full canvas cleared and filled
    // regardless of zoom/pan).
    context.clearRect(0, 0, canvas.width, canvas.height);
    context.save();
    context.fillStyle = Stairs.options.backgroundColor;
    context.fillRect(0, 0, canvas.width, canvas.height);
    context.restore();

    // Apply viewport transform — pan first, then scale around the
    // viewport centre. Centring math: canvas-centre minus
    // staircase-centre-in-local-space × zoom, plus user pan.
    context.save();
    Stairs.applyViewportTransform(context);
    Stairs.drawTreads(context, y, treads);
    Stairs.drawBallustrades(context, treads, ballustrades);
    Stairs.drawPosts(context, treads, posts);
    Stairs.drawMeasures(context);
    context.restore();
    break;
}
```

For QUARTERTURN, HALFTURN, DOUBLETURN — they already have a `save → translate → draw → restore` block. **Replace `context.translate(translateX, 0)` with `Stairs.applyViewportTransform(context)`**. The viewport transform handles centring (it accounts for the staircase's local-space origin), so the old per-case centring `translateX` math is no longer needed. Verify visually that left-direction quarter-turns still mirror correctly — the direction-driven `translateX = -translateX` flip is a separate concern and shouldn't be touched.

**New helper `applyViewportTransform`** — add near `computeFitZoom`:

```js
// Applies pan/zoom transform to the canvas context. Centres the staircase's
// local-space bounding box in the canvas, then applies user pan and zoom.
// Uniform scale only (same factor on x and y) — guarantees no skew.
Stairs.applyViewportTransform = function (context) {
    var stairWidth, stairHeight, stairOriginX, stairOriginY;

    switch (Stairs.options.type) {
        case Stairs.StairTypeEnum.REGULAR:
            stairWidth   = Stairs.options.treads.width;
            stairHeight  = Stairs.maxHeight;
            stairOriginX = Stairs.centerX - stairWidth / 2;
            stairOriginY = Stairs.options.treads.height; // top of stair stack
            break;
        case Stairs.StairTypeEnum.QUARTERTURN:
        case Stairs.StairTypeEnum.DOUBLETURN:
            // Stairs.widthPx is the exact local-space horizontal extent
            // (set in setQuarterturnStairsOptions / setDoubleturnStairsOptions).
            // Stairs.maxHeight already includes the turn flight's vertical
            // contribution (flight2Treads.width is added in setQuarterturn/
            // setDoubleturnStairsOptions). Turn flights are drawn starting
            // at local-space x = 0 — the existing centring code uses
            // (canvas.width - widthPx) / 2 as the translate, confirming
            // origin x = 0.
            stairWidth   = Stairs.widthPx;
            stairHeight  = Stairs.maxHeight;
            stairOriginX = 0;
            stairOriginY = 0;
            break;
        case Stairs.StairTypeEnum.HALFTURN:
            // Half-turn uses maxHeight1 (per the existing draw code on the
            // HALFTURN case — y = Stairs.maxHeight1, not maxHeight).
            stairWidth   = Stairs.widthPx;
            stairHeight  = Stairs.maxHeight1;
            stairOriginX = 0;
            stairOriginY = 0;
            break;
    }

    var canvasCentreX = Stairs.canvas.width / 2;
    var canvasCentreY = Stairs.canvas.height / 2;
    var stairCentreX  = stairOriginX + stairWidth / 2;
    var stairCentreY  = stairOriginY + stairHeight / 2;

    // Net translate = (canvas centre) - (stair centre × zoom) + user pan.
    var tx = canvasCentreX - stairCentreX * Stairs.viewport.zoom + Stairs.viewport.panX;
    var ty = canvasCentreY - stairCentreY * Stairs.viewport.zoom + Stairs.viewport.panY;

    context.translate(tx, ty);
    context.scale(Stairs.viewport.zoom, Stairs.viewport.zoom);
};
```

**Trace confirming the bounding-box values:**

I checked the existing draw code at lines 245, 337, 448 in Stairs.js. `Stairs.widthPx` is computed differently per turn type but represents the exact total horizontal extent in local space:

- Quarter-turn (line 245): `flight1Treads.width + treadHeight × flight2Treads.amount + SIDE_MEASURE_TAG_WIDTH`
- Half-turn (line 337): `flight1Treads.width + flight3Treads.width + treadHeight × flight2Treads.amount + SIDE_MEASURE_TAG_WIDTH × 2`
- Double-turn (line 448): `flight1Treads.width + treadHeight × (flight2.left.amount + flight2.right.amount) + SIDE_MEASURE_TAG_WIDTH`

For height, `Stairs.maxHeight` (or `maxHeight1` for half-turn) already includes the appropriate turn-flight width contribution. The existing centring math `translateX = (canvas.width - widthPx) / 2` confirms turn flights are drawn from local-space origin x = 0.

So no approximation is needed for the bounding-box — the exact values are already computed elsewhere and we reuse them.

**Direction handling.** Quarter-turn and half-turn have a `direction` config (`'left'` or `'right'`). The old code handled left direction by negating its centring translate (`translateX = -translateX`, line 522 in the QUARTERTURN case) AND by drawing flight 1 at the right edge of local space inside the draw helpers (line 649: `flight1LeftX = Stairs.canvas.width - flight1Treads.width - SIDE_MEASURE_TAG_WIDTH`). At zoom=1 the two effects compose to produce a correctly-centred mirror flip — visually equivalent to a clean `scale(-1, 1)` about the vertical centreline.

**Under our viewport transform, this composition may or may not still work.** The draw helpers' `Stairs.canvas.width` reference now sits inside a scaled context, which could cause drift at zoom ≠ 1.

For step 7 of the appendix (applying the transform pattern to QUARTERTURN/HALFTURN/DOUBLETURN), keep the existing `translateX = -translateX` direction logic when wiring up `applyViewportTransform` — don't try to fold it into the transform helper yet. After implementing, perform verification step 12 (§7): load left-direction quarter-turn, wheel-zoom to 2× and 0.5×, check for drift. Two possible outcomes:

- **No drift** → the composition works under transform; file no follow-up.
- **Drift visible** → special-case left direction in `applyViewportTransform`: `stairOriginX = Stairs.canvas.width - Stairs.widthPx` for left direction (instead of 0). This pushes the local-space origin to the right edge of canvas, mirroring where the draw helpers actually place the staircase.

This is testing-driven, not speculation-driven. The current behaviour is known-good at zoom=1; the question is purely whether scale composition preserves it.

### 3g. Inverse-scale font in drawMeasures (and variants)

The four `drawMeasures*` functions write text with `context.font = "12px " + Stairs.options.font`. Under our transform, that "12px" gets scaled by `zoom` — at zoom=2, labels render as 24px; at zoom=0.5, as 6px. To keep labels at a constant visual size regardless of zoom, divide by zoom:

```js
// Before (in each drawMeasures variant)
context.font = "12px " + Stairs.options.font;

// After
context.font = (12 / Stairs.viewport.zoom) + "px " + Stairs.options.font;
```

Four sites — one per `drawMeasures*` function (lines 2146, ~2200, ~2300, ~2400 — search for `12px`).

Same treatment for any other fixed-font sizes in the measures functions (search for `\d+px` in each function). The 12px value is the only one I spotted at a glance but there may be 10px or 14px variants for sub-labels.

Line widths also need this treatment if they should stay constant on screen — search for `context.lineWidth = ` inside the measure functions. Currently `0.5`; should become `0.5 / Stairs.viewport.zoom`. If on staging the measure lines look "too thin when zoomed in" or "chunky when zoomed out", that's the symptom — apply the same inverse-scale.

### 3h. Pointer/wheel/touch input handler

New file: `assets/js/viewport.js`. Enqueue after `Stairs.js` and before `straightFlight.js` in `baltic-wp-stairbuilder.php`.

```js
/**
 * Pan/zoom input handler for the stairbuilder canvas.
 *
 * Listens for wheel, pointerdown/move/up, and touch gesture events on the
 * canvas. Mutates Stairs.viewport.{zoom,panX,panY} and triggers a redraw
 * via Stairs.draw(Stairs.canvas).
 *
 * Uses Pointer Events API for unified mouse/touch/stylus handling.
 * Pinch gestures fall back to a manual two-touch handler since Pointer
 * Events don't emit a native pinch event.
 */
(function () {
    'use strict';

    var canvas;
    var dragStart   = null;       // { x, y, panX, panY } at pointerdown
    var pinchStart  = null;       // { dist, zoom, midX, midY } at two-touch start
    var activeTouches = new Map(); // pointerId → { x, y } for pinch tracking

    function redraw() {
        if (window.Stairs && Stairs.canvas) Stairs.draw(Stairs.canvas);
    }

    function clampZoom(z) {
        return Math.max(Stairs.ZOOM_MIN, Math.min(Stairs.ZOOM_MAX, z));
    }

    // Cursor-anchored zoom: zoom toward the point under the cursor,
    // not the canvas origin. The pan offset is adjusted so the world
    // point under the cursor stays under the cursor after zoom.
    function zoomAt(canvasX, canvasY, factor) {
        var v = Stairs.viewport;
        var oldZoom = v.zoom;
        var newZoom = clampZoom(oldZoom * factor);
        if (newZoom === oldZoom) return;

        // World point under cursor: (canvasX - panX) / oldZoom
        // We want that same world point under cursor after zoom:
        //   canvasX = worldPoint * newZoom + newPanX
        //   newPanX = canvasX - worldPoint * newZoom
        //           = canvasX - ((canvasX - panX) / oldZoom) * newZoom
        var k = newZoom / oldZoom;
        v.panX = canvasX - (canvasX - v.panX) * k;
        v.panY = canvasY - (canvasY - v.panY) * k;
        v.zoom = newZoom;
        redraw();
    }

    function onWheel(e) {
        e.preventDefault();
        var rect = canvas.getBoundingClientRect();
        var x = e.clientX - rect.left;
        var y = e.clientY - rect.top;
        // Wheel delta is +ve scrolling down (zoom out), -ve scrolling up
        // (zoom in). Scale factor of 1.1 per notch feels right; tune if needed.
        var factor = e.deltaY < 0 ? 1.1 : 1 / 1.1;
        zoomAt(x, y, factor);
    }

    function onPointerDown(e) {
        canvas.setPointerCapture(e.pointerId);
        activeTouches.set(e.pointerId, { x: e.clientX, y: e.clientY });

        if (activeTouches.size === 2) {
            // Two-finger pinch start
            var pts = Array.from(activeTouches.values());
            var dx = pts[1].x - pts[0].x;
            var dy = pts[1].y - pts[0].y;
            var rect = canvas.getBoundingClientRect();
            pinchStart = {
                dist: Math.hypot(dx, dy),
                zoom: Stairs.viewport.zoom,
                midX: (pts[0].x + pts[1].x) / 2 - rect.left,
                midY: (pts[0].y + pts[1].y) / 2 - rect.top
            };
            dragStart = null; // cancel any drag in progress
        } else if (activeTouches.size === 1) {
            dragStart = {
                x: e.clientX,
                y: e.clientY,
                panX: Stairs.viewport.panX,
                panY: Stairs.viewport.panY
            };
            Stairs.viewport.isDragging = true;
            canvas.style.cursor = 'grabbing';
        }
    }

    function onPointerMove(e) {
        if (!activeTouches.has(e.pointerId)) return;
        activeTouches.set(e.pointerId, { x: e.clientX, y: e.clientY });

        if (pinchStart && activeTouches.size === 2) {
            var pts = Array.from(activeTouches.values());
            var dx = pts[1].x - pts[0].x;
            var dy = pts[1].y - pts[0].y;
            var newDist = Math.hypot(dx, dy);
            var factor = newDist / pinchStart.dist;
            var targetZoom = clampZoom(pinchStart.zoom * factor);
            // Anchor pinch zoom at the midpoint between the two fingers
            var v = Stairs.viewport;
            var k = targetZoom / v.zoom;
            v.panX = pinchStart.midX - (pinchStart.midX - v.panX) * k;
            v.panY = pinchStart.midY - (pinchStart.midY - v.panY) * k;
            v.zoom = targetZoom;
            redraw();
        } else if (dragStart && activeTouches.size === 1) {
            Stairs.viewport.panX = dragStart.panX + (e.clientX - dragStart.x);
            Stairs.viewport.panY = dragStart.panY + (e.clientY - dragStart.y);
            redraw();
        }
    }

    function onPointerUp(e) {
        activeTouches.delete(e.pointerId);
        if (activeTouches.size < 2) pinchStart = null;
        if (activeTouches.size === 0) {
            dragStart = null;
            Stairs.viewport.isDragging = false;
            canvas.style.cursor = 'grab';
        }
    }

    function onDoubleClick(e) {
        // Reset viewport to fit-to-staircase
        Stairs.viewport.zoom = Stairs.viewport.fitZoom;
        Stairs.viewport.panX = 0;
        Stairs.viewport.panY = 0;
        redraw();
    }

    function init() {
        canvas = document.getElementById('canvas');
        if (!canvas) return;

        canvas.style.cursor = 'grab';
        canvas.style.touchAction = 'none'; // prevent browser pinch-zoom on page

        canvas.addEventListener('wheel',       onWheel,       { passive: false });
        canvas.addEventListener('pointerdown', onPointerDown);
        canvas.addEventListener('pointermove', onPointerMove);
        canvas.addEventListener('pointerup',   onPointerUp);
        canvas.addEventListener('pointercancel', onPointerUp);
        canvas.addEventListener('dblclick',    onDoubleClick);
    }

    if (document.readyState === 'loading') {
        document.addEventListener('DOMContentLoaded', init);
    } else {
        init();
    }
})();
```

## 4. What's intentionally NOT changing

- **Staircase-local drawing.** `drawTreads`, `drawPosts`, `drawBallustrades`, `drawBallustradeQuarterturn`, etc — none of these are touched. They draw in the same coordinates they always have.
- **`Stairs.options` shape.** Same fields, same structure. The form → options pipeline is unchanged.
- **The responsive sizing profiles from 1.6.4.** They still govern the initial fit calculation (a smaller staircase fits at a higher `fitZoom`). They become less visually impactful because the user can zoom freely now, but the *default* view honours them.
- **The 1.6.4 bottom-crop safety net.** Redundant in theory now (the viewport caps the visible region, not the staircase), but cheap to keep and protects against the case where `computeFitZoom` mis-fires for an unexpected geometry.
- **Quarter/half/double turn drawing.** As above — local-space drawing unchanged. The only quarter/half/double change is removing the `context.translate(translateX, 0)` line per case and letting `applyViewportTransform` handle centring.

## 5. Skew/distortion guarantee

The spec is explicit: **no skew or distortion under any zoom level.** This is guaranteed by the implementation, not just hoped for:

- `context.scale(z, z)` always passes the **same value** for x and y. We never call `context.scale(zx, zy)` with different values.
- We never use `context.transform()` or `context.setTransform()` with non-uniform matrices.
- The pan is a pure `translate` — also distortion-free.

Codify this as a comment at the top of `applyViewportTransform` (already in the snippet above). If a future maintainer is tempted to add axis-independent zoom, the comment is the warning.

## 6. Edge cases handled

- **User zooms out so far the staircase is a tiny dot.** Clamped at `ZOOM_MIN = 0.3` of fit zoom… wait, the clamp is against the *absolute* zoom value, not relative to fit. If the staircase fits at zoom=2, ZOOM_MIN=0.3 still lets the user zoom to 15% of original size. That's fine — the reset button is one click away.
- **User pans the staircase entirely offscreen.** No prevention. The reset button brings it back. We could clamp pan to keep the staircase partially in view, but it adds complexity and "user fights the clamp" UX issues. Skipping unless it actually becomes a support burden.
- **Window resize during a drag.** The pointerdown handler captures `dragStart.panX`/`panY` — if the canvas resizes mid-drag, those values are still valid in canvas-space px, so the drag continues correctly. The fit-zoom recalculates on the next `Stairs.init()` (which fires on resize via the existing ResizeObserver path).
- **Mobile orientation change.** Treated as a resize. ResizeObserver fires → `sizeCanvasToContainer` → `onLoad` → `Stairs.init` → `computeFitZoom` recomputes. User's pan/zoom persists if same staircase type. Acceptable behaviour; if it feels off in testing, we can force a reset on resize.
- **Browser without Pointer Events support.** Pointer Events shipped in all evergreen browsers years ago; the only meaningful gap is older Safari (pre-13). For those, the wheel handler still works; touch falls back to no pan/zoom. Acceptable for a stairbuilder.
- **Double-click vs double-tap on touch.** The `dblclick` event fires on touch devices when the user double-taps. We don't need a separate touch handler. Verified per MDN.

1. **Desktop, default config, straight flight.** Page renders with diagram fitting ~80% of canvas height. No page scroll beyond what's needed for the page's own content.

8. **Staircase type change resets.** Same starting state. Click the quarter-turn radio. Viewport should reset to fit-to-fit and zero pan.

10. **Measure label legibility.** At any zoom level, the "(a) XXXmm" and "(w) XXXmm" labels should look the same size on screen. If they grow/shrink with zoom, the inverse-scale in §3g is wrong.
11. **Diagnostics overlay (`?sb_debug=1`).** Viewport state line appears, updates live as user pans/zooms.
12. **Left-direction quarter-turn at non-unity zoom.** The draw helpers for turn flights reference `Stairs.canvas.width` directly (e.g. line 649) for left-direction positioning. Visually this produces a correct centred mirror-flip at zoom=1 (confirmed in pre-1.7.0 builds). Under our viewport transform, that `canvas.width` reference is now drawn inside the scaled context — which *may* cause the left-direction staircase to drift at zoom ≠ 1. **Test specifically: load left-direction quarter-turn, wheel-zoom in to 2×, verify staircase stays centred. Repeat at zoom 0.5×.** If it drifts, the fix is to special-case left direction in `applyViewportTransform` (compute `stairOriginX = Stairs.canvas.width - Stairs.widthPx` for left). If it doesn't drift, the two effects (draw-position + outer translate) compose correctly under the transform and no further action needed.

## 8. Open questions for Dan

1. **Initial viewport target — 80% confirmed; is it height or limiting-dimension?** Currently the brief uses `Math.min(fitByHeight, fitByWidth)`, i.e. whichever dimension is more constrained. So on a wide desktop canvas with a tall staircase, height wins (80% of height). On a narrow mobile canvas with a wide quarter-turn, width might win. This is the right behaviour but flagging in case you actually wanted "always height, accept horizontal overflow."
2. **The `⊕` glyph for reset.** Placeholder. Use the plugin's existing icon kit, or commission an SVG, or stay with the unicode glyph — all fine.
3. **Wheel zoom factor (1.1 per notch).** Personal preference territory. 1.1 is conventional. 1.2 feels punchier. 1.05 feels luxurious. Easy to tweak post-deploy.
4. **Pinch-zoom sensitivity.** Currently follows pinch ratio 1:1 (move fingers 2× apart → zoom 2×). Some apps damp this to 0.5× for finer control. Suggest landing at 1:1 and tweaking if it feels twitchy on staging.
